# 高可用

> **参考文档**：产品文档/高可用/YashanDB高可用概述.md

YashanDB 提供企业级高可用解决方案，通过主备复制、自动选主和多种保护模式确保业务连续性。

## 高可用部署方案

YashanDB 支持多种高可用部署模式：

| 部署模式 | 特点 | 适用场景 |
|----------|------|----------|
| 单机主备部署 | 主备库部署在不同服务器，连接同一交换机 | 本地高可用，避免单点故障 |
| 级联备部署 | 备库的备库，减少主库带宽负载 | 异地容灾 |
| 双复制组主备部署 | 两套主备环境分组，组间异步同步 | 跨地域容灾 |
| 共享集群部署（YAC） | 多实例多活部署，TAF 透明故障切换 | 集群高可用 |
| 主备集群部署 | 主备集群可部署在不同地点 | 集群容灾 |
| 存算一体分布式集群 | MN/DN 组主备部署，CN 多活 | 分布式高可用 |

## 主备复制架构

### 核心角色

**主库（Primary）**：当前提供在线数据库服务，读写模式。

**备库（Standby）**：作为数据库集群的冗余节点，支持创建最多32个备库。其核心工作机制是通过实时接收主库的redo日志，并在本地回放这些redo日志实现数据同步。

### 复制链路

```
主库 → 日志发送 → 备库 → 日志应用 → 数据同步
```

**日志发送模式**：
- **SYNC 同步传送**：从 Log Cache 读取数据发送至备库，性能较高
- **ASYNC 异步传送**：从 redo 文件读取数据发送至备库，适用于异地备库

系统自动选择 SYNC 或 ASYNC 模式。

**日志应用**：
- 备库通过回放主库发送的 redo 日志保持与主库数据一致
- 当 Log Cache 和 redo 文件中均未找到数据时，启动线程同步主库归档日志到备库

### 复制类型

- **物理复制**：将主库的物理存储内容复制到备库，使用 redo apply 机制回放。备库是主库的物理副本，数据库结构完全相同。

- **逻辑复制**：基于逻辑变更记录（SQL 语句）在备库重现数据，使用 SQL Apply 机制。逻辑备库包含与主库相同的业务数据，但物理存储结构可以不同。

### 保护模式

| 保护模式 | 事务行为 | 优缺点 |
|----------|----------|--------|
| **最大保护** | 主库事务必须等待同步备库将 redo 日志刷盘后才能返回成功 | 数据零丢失，但同步备库故障会阻塞主库 |
| **最大可用** | 主库等待 redo 日志传输到同步备库即可返回（无需等待刷盘） | 性能较高，备库故障不影响主库，可能丢数据 |
| **最大性能** | redo 写入主库文件后即可返回，无需等待备库接收 | 默认模式，存在数据丢失风险 |

> **注意**：若 COMMIT_WAIT 参数配置为 NOWAIT，主库事务提交无需等待日志落盘。建议使用默认值 WAIT。

## 主备切换

### 计划内切换（Switchover）

在备库上执行，要求主库和备库网络连接正常，且双方实例均处于 OPEN 状态。

```sql
-- 执行 Switchover（在备库上执行）
ALTER DATABASE SWITCHOVER TO primary_db_name;
```

### 故障切换（Failover）

在备库上执行，备库实例需处于 OPEN 状态。由于主库故障，redo 日志可能未完全同步，切换后可能丢失数据。

```sql
-- 执行 Failover（在备库上执行）
ALTER DATABASE FAILOVER TO standby_db_name;
```

Failover 后需要重置 redo 时间线，以区别旧主库和新主库的日志。

## 级联备部署

级联备是备库的备库——普通备库从主库接收日志，级联备从上级备库接收日志。

```
主库 → 普通备库 → 级联备
```

- 一个备库最多可连接 32 个级联备
- 级联备可以继续向下连接，无层数限制，但不可成环
- 级联备模式仅适用于单机部署
- 级联备不能执行 Switchover，只能执行 Failover

## 双复制组主备部署

在不同机房/地域部署两套主备环境，每套作为复制组：

```
GROUP 1（主复制组）    GROUP 2（备复制组）
NODE1-1(主库) ──────→ NODE2-1(异步备/首节点) ──→ NODE2-2(级联备)
    ↓                    ↓                         ↓
NODE1-2(同步备)        NODE2-3(级联备)
```

- 主复制组：主库 + 同步备库
- 备复制组：首节点（异步备）+ 级联备

## 自动选主

在单机一主多备或分布式集群节点组部署下，当主节点故障时，系统自动选举备节点升主。

### Raft 协议选主

基于 Raft 一致性算法的选举机制：

1. **任期（Term）**：每个 Leader 有任期号，正常情况下任期无限
2. **心跳机制**：Leader 周期性向 Follower 发送心跳
3. **选举超时**：Follower 在超时时间内未收到心跳则成为 PreCandidate 发起预选举
4. **投票规则**：PreCandidate 获得多数节点赞成票后成为 Candidate，发起正式选举

### Quorum 机制

YashanDB的Raft选举算法在最大保护模式下启用了Quorum机制进行选主，投票规则为：

- 节点LFN（Log Flush Number）大于等于自身，向该节点投赞成票
- 节点任期（Term）大于等于自身，向该节点投赞成票
- 满足Quorum机制的前提下，节点优先级（HA_ELECTION_PRIORITY）大于等于自身，向该节点投赞成票

选主规则为：

- QUORUM_SYNC_STANDBYS配置为MAJORITY时，Candidate需要收到多数派节点（N/2 +1）的赞成票才能成为Leader
- QUORUM_SYNC_STANDBYS配置为同步备库数量时：
  - 当syncNum >= N/2时，Candidate需要收到（N/2 +1）的赞成票才能成为Leader
  - 当syncNum < N/2时，Candidate需要收到（N - syncNum）的赞成票才能成为Leader

其中N为节点数量，syncNum为用户定义的同步备数量。

### 拓展能力

- **自动降备**：主节点未收到多数派备节点心跳响应时主动降备（HA_ELECTION_LEADER_LEASE_ENABLED）
- **自动换主**：主节点发现高优先级节点时主动降备，让高优先级节点升主（HA_ELECTION_AUTO_PRIMARY_SWITCH）

## yasom 仲裁选主

适用于单机一主一备、主备共享集群或分布式 DN 组一主一备部署。

开启仲裁选主后：
- 仅支持通过 `yasboot node switchover` 手动切换
- 主节点故障时由 yasom 自动切换，用户无法执行 Failover

```bash
# 手动切换
yasboot node switchover -c cluster.yaml -n standby_node
```

## 并行 BUILD 备库

新建一主多备或在线增加备库时，并行 build 大大提升部署效率：

- 主库同时向多台备库发送数据
- 多台备库同时执行 restore
- 级联备的并行 build 由上级备库发起

## 集群高可用

### 共享集群（YAC）

多实例采用多活部署方式：
- 客户端 TAF（Transparent Application Failover）技术实现透明故障切换
- 服务端故障自动切换能力
- 主集群多实例同时提供服务，同时向备集群传送日志

### 存算一体分布式集群

- MN 组和 DN 组采用主备部署
- 主节点不可用时从备节点自动选举升主
- CN 节点采用多活部署，单点故障不影响整体

## 运维管理

### 日常监控

```sql
-- 查看主备状态
SELECT * FROM V$PRIMARY_DB;
SELECT * FROM V$STANDBY_DB;

-- 查看复制状态
SELECT * FROM V$REPLICATION_STATUS;

-- 查看保护模式
SELECT protection_mode FROM V$DATABASE;
```

### 常见问题处理

- **备库重建**：备库故障后可重新 build
- **复制中断**：检查网络连接和日志传输状态
- **切换失败**：检查目标节点状态和日志同步情况
