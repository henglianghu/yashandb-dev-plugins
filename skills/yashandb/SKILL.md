# YashanDB

使用此技能进行安全、可量化的 YashanDB 变更。

## 工作流
1. 定义工作负载和约束条件（读写比例、延迟目标、数据量、YashanDB 版本、部署模式）。
2. 仅阅读下方各章节关联的参考文件。
3. 提出最小化改动方案，包含权衡分析。
4. 用证据验证（`EXPLAIN`、执行计划分析、锁指标、生产环境安全发布步骤）。
5. 生产环境变更需包含回滚和发布后验证。

## Schema 设计
- 写密集型 OLTP 表优先使用窄单调主键（`BIGINT` 配合 `SEQUENCE`）。
- **自增主键语法**：推荐使用 `GENERATED ALWAYS AS IDENTITY` 或 `GENERATED BY DEFAULT AS IDENTITY` 语法简化自增列实现。
- 避免使用随机 UUID 作为聚集主键；如需外部 ID，将 UUID 放在次级唯一列中。
- 字符集使用 `UTF-8`。优先使用 `NOT NULL`，时间相关列优先使用 `TIMESTAMP` 而非 `DATE`。
- 数值类型使用 `NUMBER`，变长字符串使用 `VARCHAR2`（Oracle 兼容）。

## 索引设计
- 复合索引列顺序：等值条件优先，然后是范围/排序（最左前缀规则）。
- 范围谓词会阻止后续列使用索引。
- 辅助索引隐式包含主键。长字符串使用前缀索引。
- 使用 `USING BTREE` 或 `USING HASH` 选择索引类型。
- **避免回表查询**：当查询所需的数据（SELECT列）可以直接从索引中获取时，可以避免回表操作。

## 分区策略
- 时序数据表（>5000万行）或大表（>1亿行）建议分区。提前规划——后期改造需要全表重建。
- 每个唯一键/主键都应包含分区列。始终添加 `MAXVALUE` 捕获所有。
- 支持 RANGE、LIST、HASH 分区和复合分区（RANGE-LIST、RANGE-HASH等）。

## 查询优化
- 检查 `EXPLAIN` —— 分析执行计划中的全表扫描、高成本操作。
- **重要**：需要回表时采用 TABLE FULL SCAN 效率更高，不需要回表时才采用 INDEX SCAN 效率更高。不要盲目追求索引扫描。
- 使用游标分页而非 `OFFSET`。避免在 `WHERE` 中对索引列使用函数。
- 批量插入。不需要去重时使用 `UNION ALL` 而非 `UNION`。
- 注意 Oracle 兼容语法（如 `DECODE`、`NVL`、`CONNECT BY`）。

## 事务与锁
- **默认隔离级别：`READ COMMITTED`**。YashanDB 只支持 READ COMMITTED 和 SERIALIZABLE 两种隔离级别。
- 一致的行访问顺序可防止死锁。使用退避重试死锁。
- 事务外部执行 I/O。谨慎使用 `SELECT ... FOR UPDATE`。
- **行锁特性**：行锁只有排他锁（Exclusive Lock）一种类型，不支持行级共享锁。
- **表锁语法**：使用 `LOCK TABLE ... IN EXCLUSIVE MODE`。

## YashanDB 特有功能
- **Oracle 兼容语法**：支持 DECODE、NVL、CONNECT BY 等 Oracle 语法。
- **物化视图**：用于汇总表和查询加速（仅支持 COMPLETE 刷新，不支持 FAST 刷新）。
- **存储过程**：PL/SQL 兼容的过程逻辑。**注意**：在 YashanDB 中，PL 相关功能不论在何种语法模式下均采用 yashan 语法进行解析。

## 注意事项
- 优先使用测量证据而非经验法则。
- 提供建议时注明 YashanDB 版本特定行为。

- YashanDB 与 Oracle语法高度兼容
